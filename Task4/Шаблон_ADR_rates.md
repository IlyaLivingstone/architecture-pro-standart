### <a name="_qmphm5d6rvi3"></a>**Решение**
Приведите диаграммы контекста и контейнеров в модели C4. Опишите там основные компоненты и интеграции всех элементов решения.

Для визуализации используются диаграммы модели C4:

1. [diagram_context_rates.puml](diagram_context_rates.puml) — Диаграмма контекста (C4 Level 1): показывает внешние системы.
2. [diagram_containers_rates.puml](diagram_containers_rates.puml) — Диаграмма контейнеров (C4 Level 2): детализирует поток ставок.

>  **Комментарии к диаграммам:**
> - Используется единая модель ставок: источник — АБС -> Kafka -> подписчики.
> - Для партнёра реализован SFTP-экспорт через отдельный **File Exporter**.
> - Внутренний кол-центр получает ставки через API (реализовано через кэш Redis).

Также опишите, какой логикой вы руководствовались в ходе принятия решений и выбора технологий. Не забывайте, что необходимо учесть все функциональные и нефункциональные требования.
#### Логика принятия решений

1. **Ставки хранятся в АБС**  
   - F08, +R06: централизованное управление в ядре.  
   - Решение: таблица `deposit_rates` в АБС.

2. **Рассылка через Kafka**  
   - +R01: нельзя нагружать АБС прямым опросом.  
   - Решение: при обновлении ставки АБС публикует событие в Kafka.

3. **Внутренний кол-центр — через API с кэшем**  
   - P01: нужен быстрый отклик.  
   - Решение: `Rate Cache Service` потребляет Kafka, обновляет Redis, предоставляет REST API.

4. **Партнёрский кол-центр — через файловую выгрузку**  
   - +R02: нет API, только SFTP.  
   - Решение: `File Exporter` генерирует CSV/XLSX и отправляет по расписанию.

5. **Гарантия доставки**  
   - R03: нельзя потерять обновление ставок.  
   - Решение: Kafka гарантирует доставку; File Exporter с повторными попытками.

6. **Формат файла**  
   - Простой CSV: `product_name, rate, currency, valid_from, valid_to`.

### <a name="_bjrr7veeh80c"></a>**Альтернативы**
Опишите здесь наиболее важные альтернативные решения.

| Альтернатива | Описание | Почему не выбрана |
|-------------|--------|------------------|
| **Прямой запрос из кол-центра в АБС** | Кол-центр сам опрашивает АБС по REST | Нарушает +R01: нагрузка на АБС. Не соответствует event-driven подходу. |
| **Email-рассылка файла** | Отправка файла с ставками на email партнёра | Ненадёжно, нет подтверждения получения. SFTP — стандарт для B2B. |
| **FTP вместо SFTP** | Передача без шифрования | Нарушает безопасность. SFTP — обязательный минимум. |
| **Единый микросервис для обоих кол-центров** | Один сервис делает и API, и файл | Усложняет логику. Лучше разделить по ответственности. |


**Недостатки, ограничения, риски**

Подробно опишите здесь недостатки, ограничения и риски выбранного решения.

| Риск | Описание | Митигация |
|-----|--------|---------|
| **Задержка синхронизации ставок** | До 1 минуты между АБС и кол-центром | Приемлемо, так как ставки меняются редко. SLA — 5 минут. |
| **Сбой SFTP-передачи** | Партнёр недоступен, файл не отправлен | Реализовать retry-логику (3 попытки), уведомление администратору. |
| **Ошибка в формате файла** | Партнёр не может импортировать | Чёткая документация, тестовый прогон перед продакшеном. |
| **Перегрузка Kafka** | Много событий | Мониторинг, партиционирование по продукту. Сейчас нагрузка низкая. |
