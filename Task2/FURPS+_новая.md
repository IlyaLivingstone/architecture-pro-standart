| Код  | Требования                                                                                                                                                       | Комментарий  |
|------|------------------------------------------------------------------------------------------------------------------------------------------------------------------|--------------|
| F    | Функциональные (Functionality)                                                                                                                                   |              |
| F01  | Клиент может подавать заявку на депозит через сайт, указав ФИО и номер телефона.                                                                                 | Необходима форма на сайте с валидацией данных. Интеграция с кол-центром. |
| F02  | Клиент может подавать заявку на депозит в интернет-банке, выбрав счёт и сумму.                                                                                   | Требуется интерфейс в интернет-банке. Подтверждение СМС-кодом. |
| F03  | Клиент может подавать заявку на кредит через сайт: без паспорта — предварительный контакт менеджера; с паспортом — мгновенный скоринг и предварительное решение. | При наличии паспортных данных — вызов системы скоринга онлайн. |
| F04  | В интернет-банке клиент видит персонализированные предложения по депозитам и предодобренные кредиты.                                                             | Требуется интеграция с системами скоринга и АБС для получения персонализированных условий. |
| F05  | После подачи заявки клиент получает СМС-уведомления об изменении статуса (подтверждение ставки, открытие депозита, изменение статуса кредита).                   | Использовать существующий СМС-шлюз. Требуется надёжная доставка уведомлений. |
| F06  | Менеджер кол-центра видит заявки с сайта и может связаться с клиентом для предложения особых условий.                                                            | Интеграция сайта с системой кол-центра (через АБС или промежуточную систему). |
| F07  | Менеджер в отделении продолжает работу с уже начатой заявкой (на сайте или в интернет-банке), а не создаёт новую.                                                | Требуется единый идентификатор заявки и централизованное хранение. |
| F08  | Ставки по депозитам и кредитам должны быть доступны в интернет-банке, на сайте и в АБС.                                                                          | Решение: вынести управление ставками в АБС. XLS-файлы заменить на централизованное хранение. |
| F09  | Заявки на кредит из интернет-банка и сайта должны попадать в Кредитный конвейер оперативно (в течение рабочего дня), а не раз в сутки.                           | Требует изменения механизма интеграции между фронт-офисом и Кредитным конвейером (не только по расписанию). |
| U    | Удобство использования (Usability)                                                                                                                               |              |
| U01  | Интерфейсы сайта и интернет-банка должны соответствовать корпоративной системе дизайна (цвета, шрифты, элементы).                                                | Для единообразия восприятия и снижения порога входа. |
| U02  | Процесс подачи заявки должен быть простым и занимать не более 3 шагов.                                                                                           | Для минимизации отказов пользователей. |
| R    | Надёжность (Reliability)                                                                                                                                         |              |
| R01  | Все сервисы должны быть доступны 24/7 с уровнем доступности 99,9%.                                                                                               | Особенно интернет-банк и система подачи заявок. |
| R02  | При сбое основного ЦОД интернет-банк должен автоматически переключиться на резервный.                                                                            |  |
| R03  | Сообщения (заявки, уведомления) не должны теряться при сбоях.                                                                                                    | Необходима очередь сообщений с гарантией доставки. |
| R04  | Данные клиентов (ФИО, паспорт, номер телефона) должны защищаться при передаче и хранении.                                                                        | Обязательно шифрование TLS. Хранение паспортных данных — только в защищённых системах (АБС). |
| P    | Производительность (Performance)                                                                                                                                 |              |
| P01  | Система должна выдерживать пиковую нагрузку до 10 000 одновременных пользователей.                                                                               | На этапе масштабирования. Сейчас нагрузка низкая, но нужно закладываться на рост. |
| P02  | Запросы к АБС не должны блокировать работу интернет-банка и сайта.                                                                                               | Прямые вызовы API АБС нежелательны. Решение: использование брокера событий (Kafka) и кэширования. |
| P03  | Система скоринга не должна испытывать дополнительной нагрузки от предрасчётов для всех клиентов.                                                                 | Только on-demand скоринг при подаче заявки. Предодобрения — по триггеру, а не массово. |
| P04  | Отклик всех операций должен быть меньше секунды .                                                                                                                | Особенно критично при взаимодействии с АБС (сейчас есть задержки >1 сек). Нужно кэширование справочников. |
| S    | Поддерживаемость (Supportability)                                                                                                                                |              |
| S01  | Архитектура должна позволять дальнейшее развитие (например, переход интернет-банка на микросервисы).                                                             | Закладываем модульность. Использовать события (event-driven). |
| S02  | Должна быть подробная документация по новым процессам и интеграциям.                                                                                             | Для поддержки и передачи знаний. |
| S03  | Использовать технологии, уже известные в банке: MS SQL, Oracle, Java/Kotlin.                                                                                     | Снижение рисков и затрат на обучение. |
| S04  | Для интеграций использовать Kafka (если возможно), но учесть, что текущая версия интернет-банка несовместима.                                                    | Решение: адаптер или временный REST с последующим переходом на Kafka. |
| +R   | + Ограничения (Restrictions)                                                                                                                                     |              |
| +R01 | АБС масштабируется только вертикально.                                                                                                                           | Горизонтальное масштабирование невозможно. Нельзя нагружать напрямую. |
| +R02 | Паспортные данные нового клиента нельзя хранить вне АБС.                                                                                                         | Юридическое ограничение. Требует быстрой передачи данных в АБС. |
| +R03 | Новые клиенты должны пройти идентификацию в отделении.                                                                                                           | Регуляторное требование. Онлайн-идентификации пока нет. |
| +R04 | СМС-подтверждение должно использовать существующий механизм в ядре системы.                                                                                      | Избегать доработок ядра. Реализовать логику подтверждения в рамках команды разработки. |
| +R05 | Персонализированные ставки и предодобрения должны рассчитываться без задержек.                                                                                   | Требует кэширования результатов скоринга и ставок. |
| +R06 | Все изменения должны быть совместимы с текущими платформами и иметь минимальное влияние на АБС.                                                                  | Критическое ограничение. АБС — ядро, его недоступность = остановка банка. |
